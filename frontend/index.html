About Project

                

                    This application serves as a demonstrator of an attribute-based authentication and self-sovereign identity (SSI) ecosystem that utilizes post-quantum cryptography and zero-knowledge proofs. It is based on the Signature with Efficient Protocols (SEP) scheme proposed in the paper "Practical Post-Quantum Signatures for Privacy" (Argo et al., CCS '24).

System Architecture and Contributions

                
                
Original Authors' Code:  Low-level implementation of lattice cryptography, ZKP schemes, cryptographic primitives and the mathematical aspects of lattice cryptography and basic benchmarking

                
My Contribution (Implementation):

                
                
    
                    
  • C-Kernel Architectural Separation: Decoupled the original monolithic academic code into three independent, isolated C binaries, each responsible for a distinct cryptographic phase with secure memory serialization to disk:
                        
      
                            
    • issue_credential.c (Signer): Executes the lattice key generation for the user, processes the cryptographic commitment, and mathematically computes the blind signature over the specific attributes. Outputs the raw binary credential.
      • 
                            
    • generate_proof.c (Prover): Loads the stored credential into RAM, embeds the data, and applies the Fiat-Shamir transformation to construct a mathematically sound Zero-Knowledge Proof (ZKP). Serializes this anonymous proof into an ephemeral binary file.
      • 
                            
    • verify_proof.c (Verifier): Deserializes the ZKP binary and the Authority's master public key. Performs the heavy matrix multiplications required to mathematically validate the proof's integrity without revealing the underlying attributes.
      • 
                        
    
                    
    • 
                    
    
                    
  • Python Backend Orchestrator: Designed a complete API layer built on the FastAPI framework that acts as a secure bridge to the C-kernel. It consists of:
                        
      
                            
    • authority.py (Issuer): Simulates the Trusted Issuer, handles personal information parsing, triggers C-kernel signatures, and securely logs AES-GCM encrypted audits (Data Minimization - no PII stored).
      • 
                            
    • wallet.py (Holder): Manages local secure storage. Encrypts and decrypts user attributes and ZKP binaries using AES-256-GCM.
      • 
                            
    • prover.py (Prover): Acts as the active cryptographic agent for the Wallet. It safely invokes the C-kernel to generate the ZKP and enforces local attribute validation (aborting the process locally if the user lacks the required attribute)
      • 
                            
    • verifier.py (Verifier): Orchestrates the mathematical validation of the ephemeral ZKP binary and ensures its immediate deletion to prevent replay attacks.
      • 
                            
    • gen_test_users.py An automated script that immediately populates the test environment with five predefined users, fills in their personal information, generates individual attribute-based credentials for each user, and stores them in the wallet. Access the wallet using the password: test
      • 
                        
    
                    
    • 
                    
    
                    
  • Frontend User Interface (index.html): A lightweight, dependency-free single page application built with Vanilla JavaScript, HTML5, and CSS3. It acts as the visual and orchestration layer for the entire demonstrator, providing:
                        
      
                            
    • Tabbed navigation seamlessly simulating the workflow between the Issuer (data entry, credential generation, encrypted audit logs), the Wallet (secure local storage, identity management), and the Verifier (scenario-based zero-knowledge proofs
      • 
                            
    • Utilizes asynchronous JavaScript (Fetch API) for seamless communication with the Python backend without page reloads.
      • 
                            
    • Provides Real-Time Telemetry Dashboard
      •

Benchmarks

                
Tab displays real-time telemetry extracted directly from the isolated C-kernel executions using CLOCK_MONOTONIC.

                
Issuer

                
    
                    
  • User Keygen: Time to generate the user's secret/public key pair.
    • 
                    
  • Crypto Commitment: Time to securely commit the user's attributes into a lattice matrix.
    • 
                    
  • Blind Signature: Time taken by the authority to mathematically sign the committed attributes without seeing the actual data.
    • 
                    
  • Phase Sign (Issuance): Total duration of the issuance protocol (sum of the above steps)
    • 
                    
  • Single Credential Size: Disk space consumed by one binary lattice credential.
    • 
                    
  • Total Wallet Size: Combined size of all 3 issued credentials (Age, License, Record).
    • 
                


                
Prover

                
    
                    
  • Wallet & Keys Load (I/O): Time required to load lattice keys from the disk into RAM.
    • 
                    
  • Data Embedding: Time to embed the blind signature into the ZKP polynomial ring.
    • 
                    
  • Fiat-Shamir Execute: Time to execute the interactive-to-non-interactive transformation (the core proof generation).
    • 
                    
  • Phase Gen (Prover): Total execution time required to generate the anonymous proof.
    • 
                    
  • Total Proof Payload Size: Total size of the generated binary payload (Proof + CRS + UPK seed).
    • 
                    
  • Pure ZKP Data Size: Size of the mathematically blinded zero-knowledge vectors alone
    • 
                


                
Verifer

                
    
                    
  • RAM Setup (Allocation): Time to allocate memory for FLINT matrix structures.
    • 
                    
  • PK & Proof Load (I/O): Time to deserialize the proof and the Authority's Master Public Key.
    • 
                    
  • Mathematical Verify: Time spent computing polynomial multiplications to check validity.
    • 
                    
  • Phase Verify (Verifier): Total execution time of the Verifier entity.
    • 
                    
  • Authority Public Key Size: File size of the Authority's Master Public Key.
    • 
                    
  • Peak RAM Usage: Maximum Resident Set Size (memory footprint) during the verification process.

User Manual

                
The application is designed as a single-page application and is divided into five logical sections

                
                
Tab Isuuer

                
    
                    
  • Form: Used to enter the user’s personal information (first name, last name, ID card number, date of birth, address) and define their attributes for cryptographic signing (age of majority, driver’s license, criminal record).
    • 
                    
  • Wallet Encryption Key: Here, the user defines the master password that will be used to locally encrypt their wallet using the AES-256 algorithm.
    • 
                    
  • Audit Registry: Displays an encrypted transaction log.  It contains only a timestamp, an anonymous ID, and a list of signed attributes, but does not contain any personal data
    • 
                    
  • The "Generate Test Database" button allows you to instantly populate the system with five predefined test profiles for a quick demonstration of various scenarios. The password to enter in the wallet is: test
    • 
                


                
Tab Users Wallet

                
    
                    
  • Secure Storage Locked: The wallet is locked upon opening. A password defined at issuance is required to access the data (for test users, the password is "test")
    • 
                    
  • Credential Store: After unlocking, identity cards are displayed. Each card clearly lists personal data and the status of cryptographic attributes.
    • 
                    
  • The “Select Identity for ZKP Verification” button initiates the verification protocol (Prover) for the selected identity.
    • 
                    
  • Lock Session: Allows the user to immediately delete decrypted data from the browser’s operating memory and lock the wallet again
    • 
                



                
Tab Verifier

                
    
                    
  • Drop-down menu for selecting the type of check: Age Verification (Age Verification > 18); Police Check (Driver's License Validity Check); HR Screening (Criminal Background Check).
    • 
                    
  • Verification Result: Once the protocol has run, the verdict “AUTHORIZED” (in green) or “UNAUTHORIZED” (in red) will be displayed, along with the subject’s anonymous ID and confirmation of the mathematical validity of the proof
    • 
                


                
Tab Benchmarks

                
    
                    
  • Panel showing performance metrics by time and size. A detailed breakdown of metrics organized into the Issuer, Prover, and Verifier phases. This panel tracks computation times, I/O operations (disk reads), RAM allocation, binary proof sizes, and peak memory usage in MB.
    • 
                    
  • A more detailed description of the individual metrics is available on the "About" tab
    •  
                


                
Tab About

                
    
                    
  • It includes technical documentation for the project, a description of the system architecture, a list of technologies used (C, Python, JavaScript), and an explanation of the measurement methodology in relation to the reference paper.
